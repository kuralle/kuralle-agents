Now, before we go into actually making a skill ourselves and showcasing that, what I want to address quickly are the differences and similarities between agent skills, function calling, and the model context protocol. With all of these different standards and ways to get models and agents to perform actions, call tools, and do other things. I've seen a little bit of confusion as to where exactly each one of these concepts play. So, let's quickly break down the outline of all of these concepts so that we can show how they work together. Starting with function calling, we understand function calling as a way to enable language models to essentially invoke APIs, take different actions or use different tools that have been defined. The models themselves are not actually triggering or sending any sort of request. Rather, they are provided a schema of a tool and then at its generation step can reason that it wants to use or call these tools and generate the arguments for that specific schema. The whole process then follows a little bit something like this. A developer is going to define the schema of whatever external tool or API it wants the model to be able to use. Different metadata about the tool and the schema is then going to be packaged along with the system prompt to be passed to the model. This often includes things like the explanation and description of the tool and maybe some of the descriptions of the individual arguments and what is expected. Then at runtime, the language model given whatever input from a user from a system it receives can choose to use an available tool by outputting the corresponding tool called schema that it understands as well as which tool it wants to use and generating the various arguments for it. Whatever client or application is then supporting the language model here is going to identify that a tool call has been generated, parse out all of those arguments and then execute that tool in whatever environment has been developed. The results of that tool are going to be returned back to the model in a formatted tool message for the model to observe and make a subsequent follow-up generation based on. And while this is the foundation of how tools are called and used within various agents, the issue that kind of came about when function calling was created is actually this tool execution and surfacing part where essentially every single application had a different defined format and handling of tools which is why the model context protocol was actually created. The model context protocol or what is mostly referred to as MCP is essentially a server that standardizes the execution of and formatting of these tools to the models. So the process that MCP follows is that the developer is first going to create an MCP server that defines the tools schema and execution logic following this standard. then the application or client that the language modeler agent is operating within is going to be built to support the MCP standard. And so what it'll do is it will connect to that MCP server and be able to surface or expose the various different tools in the same manner to the model. Then very similarly to just regular function calling, the language model when it chooses to use or invoke a tool is going to generate the corresponding MCP friendly schema as well as all of the arguments. The client will then identify that and pass those arguments to the server. The server itself is then going to handle the execution of the tool and provide back the results which are then appended on as a tool message for the language model to observe and take some sort of subsequent action or guide its further generation. So MCP is best used for and most often known for standardizing the surfacing and execution of these tools. Where in the prior format each implementation of tools would have to have its own custom logic for actually executing and returning. MCP provides the foundation and standard for this. Now MCP doesn't just handle tools. They also have a way to surface prompts and additional resources when defined. But I haven't seen this be adopted as widely as the tool called standardizing has been. But that is all right because I think a better implementation of this and my preferred method of surfacing different resources and prompts and surrounding context is via agent skills. In comparison, agent skills follows the process where a developer creates the skill repository with a primary skill markdown file and all of the different resources, assets, scripts, and anything related to that skill all in one nice directory. Then, similar to how the tool call outlines and schemas are passed as context to the model, a supporting file system-based aging client will discover what skills are available and load in just the name and description into the prompt or as some component of the language models instructions. At runtime, the language model and the agent here can choose to reference an available skill by navigating to the corresponding directory and loading the primary skill markdown file using its navigation and file management tools. This then loads all of the contents and essentially injects all of this information into the language model context where the language model can then use to follow all these provided instructions, load additional information or run different scripts that are packaged alongside the skill and take any further actions or guide its subsequent generation. here. Any of these actions can be formatted as a tool call ran through a MCP server or use the existing tools that are within the file system based agent harness. Things like running a bash command to execute a script might be an example there. But as you can see, these three concepts of function calling, MCP, and agent skills aren't necessarily in competition with each other. Rather, they build on each other to enable different levels of things. function calling for using different tools and generating arguments. MCP to standardize those function callings and agent skills to bundle all of that together for specific workflows or behaviors.
