# Harness Engineering (OpenAI) — source capture

Source: https://openai.com/index/harness-engineering/ (fetched 2026-06-07 via jina reader)

---

Over the past five months, our team has been running an experiment: building and shipping an internal beta of a software product with **0 lines of manually-written code**.

The product has internal daily users and external alpha testers. It ships, deploys, breaks, and gets fixed. What's different is that every line of code—application logic, tests, CI configuration, documentation, observability, and internal tooling—has been written by Codex. We estimate that we built this in about 1/10th the time it would have taken to write the code by hand.

**Humans steer. Agents execute.**

We intentionally chose this constraint so we would build what was necessary to increase engineering velocity by orders of magnitude. We had weeks to ship what ended up being a million lines of code. To do that, we needed to understand what changes when a software engineering team's primary job is no longer to write code, but to design environments, specify intent, and build feedback loops that allow Codex agents to do reliable work.

## We started with an empty git repository

The initial scaffold—repository structure, CI configuration, formatting rules, package manager setup, and application framework—was generated by Codex CLI using GPT-5, guided by a small set of existing templates. Even the initial AGENTS.md file that directs agents how to work in the repository was itself written by Codex.

Five months later, ~1M lines of code, ~1,500 PRs merged with 3 engineers (now 7), avg 3.5 PRs/engineer/day, throughput *increased* with team size. Core philosophy: **no manually-written code**.

## Redefining the role of the engineer

The lack of hands-on human coding introduced a different kind of engineering work, focused on systems, scaffolding, and leverage.

Early progress was slower than expected, not because Codex was incapable, but because the **environment was underspecified**. The agent lacked the tools, abstractions, and internal structure required to make progress. The primary job of the engineering team became enabling the agents to do useful work.

Working depth-first: breaking larger goals into smaller building blocks (design, code, review, test), prompting the agent to construct those blocks, and using them to unlock more complex tasks. When something failed, the fix was almost never "try harder." The question was always: **"what capability is missing, and how do we make it both legible and enforceable for the agent?"**

Humans interact almost entirely through prompts. To drive a PR to completion: instruct Codex to review its own changes locally, request additional specific agent reviews both locally and in the cloud, respond to human/agent feedback, and iterate in a loop until all agent reviewers are satisfied (a **Ralph Wiggum Loop**). Codex uses standard dev tools directly (gh, local scripts, and **repository-embedded skills**) to gather context.

## Increasing application legibility

As code throughput increased, the bottleneck became human QA capacity. They added capabilities by making the application UI, logs, and app metrics directly legible to Codex.

- Made the app **bootable per git worktree**, so Codex could launch and drive one instance per change.
- Wired the Chrome DevTools Protocol into the agent runtime; created **skills** for working with DOM snapshots, screenshots, and navigation. Codex could reproduce bugs, validate fixes, reason about UI.
- Observability: logs/metrics/traces exposed via a local observability stack, ephemeral per worktree. Agents query logs with LogQL and metrics with PromQL. Prompts like "ensure service startup completes in under 800ms" become tractable.

Single Codex runs work on a single task for upwards of six hours.

## We made repository knowledge the system of record

Context management is one of the biggest challenges. Earliest lesson: **give Codex a map, not a 1,000-page instruction manual.**

- **Context is a scarce resource.** A giant instruction file crowds out the task, the code, the relevant docs.
- **Too much guidance becomes non-guidance.** When everything is "important," nothing is.
- **It rots instantly.** A monolithic manual becomes a graveyard of stale rules.
- **It's hard to verify.** A single blob doesn't lend itself to mechanical checks.

So instead of treating `AGENTS.md` as the encyclopedia, they treat it as **the table of contents**.

The knowledge base lives in a structured `docs/` directory treated as the system of record. A short `AGENTS.md` (~100 lines) is injected into context and serves primarily as a **map**, with pointers to deeper sources of truth.

Plans are first-class artifacts. Ephemeral lightweight plans for small changes; complex work captured in execution plans with progress and decision logs checked into the repo. Active plans, completed plans, and known technical debt are versioned and co-located.

This enables **progressive disclosure**: agents start with a small, stable entry point and are taught where to look next.

Enforced mechanically: dedicated linters and CI jobs validate the knowledge base is up to date, cross-linked, structured correctly. A recurring **"doc-gardening" agent** scans for stale docs and opens fix-up PRs.

## Agent legibility is the goal

Because the repository is entirely agent-generated, it's optimized first for **Codex's legibility**. From the agent's point of view, **anything it can't access in-context while running effectively doesn't exist.** Knowledge in Google Docs, chat threads, or people's heads is not accessible. Repository-local, versioned artifacts (code, markdown, schemas, executable plans) are all it can see.

Favored dependencies and abstractions that could be fully internalized and reasoned about in-repo. "Boring" technologies are easier for agents to model (composability, API stability, training-set representation). In some cases cheaper to have the agent reimplement subsets of functionality than work around opaque upstream libraries (e.g. own map-with-concurrency helper instead of `p-limit`).

## Enforcing architecture and taste

**By enforcing invariants, not micromanaging implementations, agents ship fast without undermining the foundation.** e.g. require Codex to parse data shapes at the boundary, but not prescriptive on how (model likes Zod).

Built the app around a rigid architectural model. Each business domain divided into a fixed set of layers, with strictly validated dependency directions: **Types → Config → Repo → Service → Runtime → UI**. Cross-cutting concerns (auth, connectors, telemetry, feature flags) enter through a single explicit interface: Providers. Enforced mechanically via custom linters and structural tests.

"Taste invariants": statically enforce structured logging, naming conventions, file size limits, platform-specific reliability. Custom lints inject remediation instructions into agent context via error messages.

Enforce boundaries centrally, allow autonomy locally. Output need not match human stylistic preferences as long as it's correct, maintainable, and legible to future agent runs.

## Throughput changes the merge philosophy

Minimal blocking merge gates. PRs short-lived. Test flakes addressed with follow-up runs rather than blocking. In a system where agent throughput far exceeds human attention, corrections are cheap and waiting is expensive.

## What "agent-generated" actually means

Agents produce: product code and tests, CI config and release tooling, internal dev tools, documentation and design history, evaluation harnesses, review comments and responses, scripts that manage the repository itself, production dashboard definition files.

## Increasing levels of autonomy

Given a single prompt, the agent can: validate current state, reproduce a reported bug, record a video of the failure, implement a fix, validate by driving the app, record a second video, open a PR, respond to feedback, detect/remediate build failures, escalate to a human only when judgment is required, merge.

## Entropy and garbage collection

Full autonomy introduces novel problems. Codex replicates existing patterns—even uneven ones—leading to drift. Initially humans spent every Friday (20% of the week) cleaning "AI slop." Didn't scale.

Instead, encoded **"golden principles"** directly into the repo + built a recurring cleanup process. Opinionated, mechanical rules. e.g. (1) prefer shared utility packages over hand-rolled helpers; (2) don't probe data "YOLO-style"—validate boundaries or rely on typed SDKs. Background Codex tasks scan for deviations, update quality grades, open targeted refactoring PRs, automerged. Functions like **garbage collection**. Technical debt is a high-interest loan—pay it down continuously.

## What's become clear

Building software still demands discipline, but the discipline shows up more in the **scaffolding** rather than the code. The tooling, abstractions, and feedback loops that keep the codebase coherent are increasingly important. Most difficult challenges now center on **designing environments, feedback loops, and control systems**.
