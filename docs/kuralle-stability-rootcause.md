# Why Kuralle's primitives lack stability for human-like, always-on conversation

## Verdict

Kuralle is architected as a **synchronous, single-session, generation-and-control-fused turn machine** — the model speaks and the orchestrator routes inside the same blocking loop, one input per turn, with the provider owning turn-taking and tool failures crashing the turn outright. Every production bar that defines a 100%-on-call agent (ElevenLabs' generation/control separation and speculative endpointing, Fin's confidence-gated validation, the shopping agents' durable cross-surface memory and guarded autonomy) assumes the *opposite* posture: a thin orchestrator that runs prediction, routing, guardrails, and recovery as **parallel layers outside the prompt**, never blocking the user-facing reply. The instability is not a set of bugs — it is one architectural inversion (control living *inside* generation instead of *around* it) expressed across seven dimensions, with escalation/tool-failure being the only place it has already become a hard crash rather than a soft ceiling.

## Cross-cutting themes

**1. Control is fused into generation instead of running around it (the keystone).** Kuralle's defining choice is the unified turn loop: one prompt (`composeSystem`), one tool dict, the model speaks, *then* `node.next()` / `selectHostTarget` / `classifyControl` interpret what happened (`runFlow.ts:136-138`, `hostLoop.ts:45-63`, `classifyControl.ts`). ElevenLabs' entire stability claim is the inverse — generation and control are *separate parallel layers*, so transition logic, guardrails, and tool dispatch never contaminate the prompt and never block the reply. This single inversion is the upstream cause of ranks 2, 3, 4, 6, and 7: with no out-of-band evaluator there is nowhere to put turn-end prediction, speculative generation, mid-flow rerouting, per-node guardrails, or confidence gating except *inside* the blocking generation step, where they don't fit. Every "Kuralle has no primitive for X" in the diagnoses resolves to "X belongs in a control layer Kuralle doesn't have."

**2. The turn is a blocking, sequential, single-input checkpoint.** `awaitUser()` (`runFlow.ts:193`), the single-slot input buffer, `turnInputConsumed`, and the `await`-chained `hostLoop.ts:177-184` all encode the same assumption: exactly one user input arrives, is consumed once, fully processed, then control returns. This makes the framework deterministic and easy to reason about (its real virtue — `runSilentExtraction` and the consumed-once gate genuinely prevent hallucinated field fabrication) but it is structurally incapable of the always-on behaviors: speculative pre-generation, background listening, multi-intent buffering, async tool fire-and-forget, and latency isolation all require *concurrency the turn model forbids*. The provider-owned turn-taking (rank 3) is a symptom: Kuralle delegated endpointing precisely because its own loop has no place to run a predictor concurrently.

**3. State is bound to one in-process session, not to a durable user.** `openRun` loads by exact `sessionId`, memory is opt-in and silently degrades without `userId`, and there is no scheduler, no resumption checkpoint, no proactive trigger. The shopping-agent bar (durable cross-device memory, unattended scheduled actions, guarded autonomy) is simply outside Kuralle's current ontology — sessions are passive and live-only. This theme is narrower than the other two (it affects rank 5 primarily) but it is the one that most clearly separates "a good conversation engine" from "a 100%-on-call agent."

## Top 3 highest-leverage CORE changes

### 1. Split orchestration from generation: a parallel control layer

**What to change.** Extract transition evaluation, guardrails, and intent routing out of the generation path. The agent turn generates from a *minimal, node-scoped* prompt (node goal + node-scoped KB + node-scoped tools only); a separate evaluator — running concurrently or immediately after, but never feeding the agent's prompt — scores transition conditions, runs guardrail classifiers, and dispatches the next node. Concretely: stop merging all tools into one `aiTools` dict (`TextDriver.runAgentTurn:59-80`), stop concatenating control hints into `composeSystem` (`nodeBuilders.ts:30-37`), and move `node.next()` / `selectHostTarget` / `classifyControl` into a control component that the driver calls without the model seeing it.

**Why it closes multiple gaps.** This is the keystone for theme 1. Once a control layer exists outside the prompt, it becomes the natural home for: turn-end prediction (rank 3), speculative generation scheduling (rank 4), mid-flow digression rerouting (rank 6, by letting `selectHostTarget` run even when `activeFlow` is set), per-node guardrails and confidence gating (rank 7), and the recover/escalate decision (rank 1). Five high-severity dimensions collapse into "implement them in the layer this change creates."

**Flip condition — do NOT do this if** Kuralle's target stays text-first, single-turn, deterministic flows (the hospital-demo / Acme shape) where the consumed-once determinism is the selling point and latency masking is irrelevant. A parallel evaluator adds real complexity and a second model call per turn; if no consumer needs voice-grade responsiveness or mid-flow digression, this is over-engineering against §2 (Simplicity First).

### 2. A turn-local recovery boundary + a "cannot-do/escalate" control result

**What to change.** Wrap `ctx.tool()` execution in the turn loop (`TextDriver.ts:88-145`, `VoiceDriver` background path) in a recovery boundary that catches the exception instead of letting it abort the session (`Runtime.run:176-236`). Add control result types to `classifyControl.ts` beyond `__handoff`/`final`: a `tool_failed`/`cannot_do` result the executor produces on throw, and an `escalate` result the model or runtime can emit — making the declarative `escalate` transition (`flow.ts:183-186`) reachable at runtime. On failure, the turn composes an honest in-turn message ("I hit a problem doing X — let me get a person") and routes to the human terminal target rather than crashing.

**Why it closes multiple gaps.** Directly fixes the only *critical* dimension (rank 1) and is a prerequisite for the shopping-agent trust bar (rank 5): an always-on agent authorized to take real actions must degrade gracefully, never leave an orphaned session. It also gives the control layer (change 1) a concrete failure-handling surface, so the two compose.

**Flip condition — do NOT do this if** tool failures are genuinely meant to be fatal and externally supervised (e.g. a batch/back-office context where a crash + alert + manual replay via the `RunStore` step log is the desired operational model). In that case, swallowing exceptions into a conversational fallback would *hide* failures that operators want surfaced loudly.

### 3. Per-node context scoping (KB + memory + query) as a first-class node field

**What to change.** Add optional `knowledge`/`memory` fields to `ReplyNode` (and collect/decide) in `flow.ts:33-40`, and make `runGatherPhase` (`TextDriver.ts:46-51`) assemble context *per node* from those fields rather than once agent-wide. Route retrieval through a node-specific query (a per-node query-rewrite over history) instead of the raw `latestUserMessage` (`knowledge.ts:54`). Tools are already per-node-resolvable; normalize KB/memory to match so each node sees only what its goal needs.

**Why it closes multiple gaps.** Smaller per-node context is ElevenLabs' stated reliability mechanism — it directly addresses grounding/hallucination (rank 7) and meaningfully helps determinism (rank 2: a tightly scoped node prompt is harder to derail). It also unlocks the per-node tool/KB binding the digression evaluator (rank 6) needs to know what each node *can* handle. It is the lowest-risk of the three: additive optional fields, no behavioral change for agents that don't set them (§3 Surgical Changes).

**Flip condition — do NOT do this if** agents are small and single-goal (one flow, a handful of nodes, one KB). Per-node scoping pays off when the agent has many nodes with divergent goals and a large KB where cross-node bleed causes wrong retrievals; for a tiny agent it adds authoring burden and config surface for no reliability gain.
